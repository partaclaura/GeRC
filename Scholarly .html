Introduction in GeRC

We wanted to create a tool under the form of a software product which should help help users have more dynamic lifestyles, starting with their home appliances. Under the premise that more and more people in Romania are starting to embrace the idea of remote control by buying smart appliances, we create an environment to have all of them connected and in one place.

The product is designed to be supported on 2 platforms: desktop and mobile for better flexibility. On these platforms, users can interact using one of the following 3 paradigms:

  1. Keyboard (desktop + mobile): the keyboard is only required on desktop and mobile for the login part, afterwards the keyboard is avoided;
  2. Mouse (desktop): only clicks are required to interact with the product, keeping it simple and similar to gesture (mobile) interactions;
  3. Gesture (mobile): just tapping the buttons should suffice to roam in the mobile application, but device's on / off sliders could be swiped instead;
  4. Vocal (mobile): when users are on the mobile version, they can activate the voice assistant with a voice command and then vocalize his intented action (ex.: turn on / off a certain device).


Interaction Design Patterns

  • There are some intrinsic difficulties to be expected along the way, as in the fear of technology (people who love the idea, but would be afraid that it would behave unexpectedly) or having too many devices and/or settings on each device which are increasingly difficult depending on overall experience and number of applications previously required. Therefore, there are more than several tentatives of solution from where we could take inspiration, but the result would be an actual combination of them as keeping the user's needs in mind remains the highest priority.
  • The software is intended to be used as a master command for home appliances. The main actions to be taken from the application would be adding / removing devices, turning them on / off and changing gestures which control them.
  • Our product is designed for daily use, for short periods of time, keeping the user interaction simple and easy to use. At the same time, we have taken into consideration portability by making the software in two versions: desktop and mobile.
  • As presented in the persona documents, the intended audience falls under the umbrella of tech savvy individuals, busy parents and people working prolonged periods of time away from home.
  • Regarding the software's usability, we have taken into consideration: learnability (suggestive icons for buttons), efficency (avoidance of overly complicated menus), errors (an user cannot make mistakes except for login section) and satisfaction (the design is simple, cute, suggesting repetitive uses).
  • Practical examples:
    • Mobile: top bar (navigation), grids (layout), tap (actions and commands), tables as lists (tables and lists) and label, input, active, textarea, slider, single-select, password, email, pop-up on-screen integrated keyboard (form patterns);
    • Desktop: top bar (navigation), bottom bar (navigation - help), grids (layout), tap (actions and commands), tables as lists (tables and lists) and label, input, active, textarea, slider, single-select, password, email (form patterns);
    • Vocal: vocal commands (actions and commands), dedicated screens (form patterns - voice screen).

User Flow

User flow for first persona:

  1. Login screen: submit valid username and password. It also has the posibility to sign up or reset the password in case it is forgotten.
  2. Add device (if not already existing). Only required once per device unless the user decides to unpair and pair again the device with the software. Any device connected to te local Wi-Fi is available for pairing.
  3. Select device to control. Each device has a pictogram representing itself, with two possible statuses (on and off) for easier identification.
  4. Turn device on / off, change device settings, remove device. In the use case of turning the device on / off, the application would refresh the device screen with the new status. This is easy to identify by the usual left-right slider.

User flow for second persona:

  1. Login screen: submit valid username and password. It also has the posibility to sign up or reset the password in case it is forgotten.
  2. Add device (if not already existing). Only required once per device unless the user decides to unpair and pair again the device with the software. Any device connected to te local Wi-Fi is available for pairing.
  3. Select device to control. Each device has a pictogram representing itself, with two possible statuses (on and off) for easier identification.
  4. Enter change gesture menu. Each gesture also has a pictogram which makes it easy for the user to consider pressing in order to modify.
  5. Select which user's gestures should be modified (if needed). The currently selected user, corresponding by default to the currently logged user, is highlighted on the top of the screen. The user then can choose whose gesture to modify (if the account plan allows him to do so, similar to local admin rights, on appliances level).
  6. Pick the new gesture. As upper mentioned, the gestures themselves are easy to understand and identify, but on the gestures screen, they can also be highlighted with green (for the already selected gesture on that action) or red (for gestures already selected but on different actions).
  7. Save / cancel changes. At the bottom of the screen are the usual save (green) and cancel (red highlight) buttons which will redirect the user to the previous screen (device screen). The cancel action can also be triggered by pressing the back and home buttons at the top of the screen.

Affective Factors & Emotion

    Aesthetics

    The layouts, graphics and icons used in the software are big, rounded and cartoonish drawn in order to create a visually pleasing appealing interface. The colors are bright, vibrant, neutral for the theme blue and the typography is short and simplistic to contribute to the overall emotional tone of the interface, making it firendlier.

    Consistency

    The consistency is a very relevant factor, as the software is intended for daily use (turning devices on / off), therefore creating a mental mapping of the app and making use of the muscle memory are key. In this sense, the workflows are similar, they have very little to no inconsistencies regarding the steps taken, even for different device usage (web vs. mobile).

    Branding and Personality

    The product's personality relies on the theme colors, especially the blue, present on each and every case, combined with clean lines and simple pictograms create a branding balanced between professional and childish.

    Feedback and Microinteractions

    Feedback comes very important in the software's features, considering that remote control actions are not required to be seen by user itself at the moment of interaction. Therefore, adding colorful microinteractions and visual redirects become crucial in avoiding the user to turn the device on / off numerous times thinking it didn't trigger anything.

    Emotional Context

    The emotional context in which the software will be used falls more under the tool apps context, where the design should be simple and efficient. This factor brings the product closer to the objective of daily usage.

    User Empathy

    Considering that empathy can be an crucial aspect of emotional design, we have tested and interacted with several already existing solutions in order to find their strengths and especially their weaknesses and apply / fix them into GeRC.
